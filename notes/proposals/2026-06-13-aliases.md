# Proposal: Aliases / Redirects

*2026-06-13*

## Motivation

Digital gardens are renamed and reorganized constantly — that's the whole point
of growing notes over time. Italic already makes *internal* moves painless:
wikilinks resolve by stem, so moving a file doesn't break links inside the
garden. But the moment a note has been shared, bookmarked, or indexed by a search
engine under its old URL, moving it produces a 404 for the outside world.

Italic has no answer to this today. Hugo solves it with `aliases:` frontmatter
that generates redirect stub pages at the old locations. This proposal adds the
same: a per-document `aliases` list that emits tiny redirecting HTML pages
pointing at the document's canonical URL.

This was flagged as the **highest-priority Tier 1 gap** in
[the Hugo gap analysis](2026-06-13-gap-analisys.md) — it's the most
garden-specific missing feature, and it's small.

## Prior art: Hugo

Hugo's model (which we'll mostly mirror, since users migrating from Hugo/Obsidian
publish tooling will expect it):

```yaml
---
title: My Reorganized Post
aliases:
  - /old-url/
  - /2019/an-even-older-url/
  - /posts/legacy.html
---
```

Each alias becomes an HTML file containing a `<meta http-equiv="refresh">`
redirect, a `rel="canonical"` link, and a JS fallback, pointing at the page's
real permalink. Hugo appends `index.html` when the alias has no file extension,
and writes the literal path when it does.

## Design

### Frontmatter

A new optional frontmatter field, `aliases`, holding a sequence of strings:

```yaml
aliases:
  - /old-url/
  - /posts/legacy.html
```

- Read during `Doc::uplift_frontmatter` (src/doc.rs) into a new
  `pub aliases: Vec<String>` field on `Doc`, the same way `template`/`terms` are
  uplifted. Non-string entries are skipped (mirror `uplift_terms`' tolerance).
- Participates in **collection defaults** for free: because `apply_defaults`
  merges into `self.data` and the pipeline re-uplifts afterward, a collection can
  set a default `aliases` pattern — though list-valued defaults are an unusual use
  and we needn't document it prominently.
- Because aliases are read off the same `Doc`, drafts get the correct behavior for
  free: draft docs are dropped at the read phase, so their aliases are never
  emitted.

### Alias path → output path

Each alias string is normalized into an output path using the **same trailing
convention italic already uses for permalinks**, extended with Hugo's
extension rule so both `/old/` and `/old` Just Work:

1. Strip a leading `/` (consistent with `permalink::expand`).
2. If the alias ends in `/` **or** has no file extension → append `index.html`.
3. Otherwise treat it as a literal file path.

Examples:

| `aliases:` entry      | Output file            |
|-----------------------|------------------------|
| `/old-url/`           | `old-url/index.html`   |
| `/old-url`            | `old-url/index.html`   |
| `/posts/legacy.html`  | `posts/legacy.html`    |
| `/feed.xml`           | `feed.xml`             |

This is a new helper, `permalink::alias_output_path(alias: &str) -> PathBuf`,
sitting next to `expand`/`default_for`/`to_url` in src/permalink.rs. Note it does
**not** do `:slug`/`:yyyy` token expansion — aliases are literal historical URLs,
not patterns. (Keeping it literal also avoids a footgun where a stray `:` in an
old URL gets rewritten.)

### Redirect target (the canonical URL)

The target is the document's own URL, computed exactly as the `link` filter does
today: `base_path` + `permalink::to_url(doc.output_path)`. That keeps redirects
correct under a sub-path deployment (`site.base_path: /blog`). When `site.url` is
set we additionally emit an absolute `rel="canonical"`, which is what search
engines actually consolidate on.

### Redirect HTML

Default built-in template (kept deliberately close to Hugo's `_internal/alias.html`
so it behaves identically in browsers and crawlers):

```html
<!doctype html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <title>{{ title }}</title>
  <link rel="canonical" href="{{ target_absolute_or_relative }}">
  <meta http-equiv="refresh" content="0; url={{ target }}">
</head>
<body>
  <p>This page has moved to <a href="{{ target }}">{{ target }}</a>.</p>
  <script>location.replace({{ target_json }} + location.hash);</script>
</body>
</html>
```

Notes:
- `rel="canonical"` is what gets crawlers to consolidate the old URL onto the new
  one. We deliberately **do not** add `robots: noindex`, matching Hugo's default
  behavior (a `noindex` would tell crawlers to drop the page entirely rather than
  pass its ranking to the canonical target).
- The JS `location.replace(... + location.hash)` preserves deep-link fragments
  (e.g. someone bookmarked `/old/#a-heading`), which the `<meta refresh>` alone
  drops. Small touch that matters for a notes site full of heading anchors.

**Override:** if `templates/alias.html` exists (resolved through the normal theme
overlay via `template_roots()`), it's used instead, rendered with a small context
(`{ target, target_absolute, title, page }`). Otherwise the built-in string
template is used. This mirrors how Hugo lets you override `layouts/alias.html`,
and means a theme can ship its own.

### Pipeline placement

Aliases need each doc's **final** `output_path` (post-defaults, since `permalink`
can arrive via collection defaults), but they don't need rendered bodies. That's
exactly the position archives occupy. Add a sibling stage:

```
4. markup ║
5. classify::taxonomies / backlinks      → freeze into Arc<DocIndex>
6. archive ║        → Vec<(PathBuf, String)> view pages
6b. alias           → Vec<(PathBuf, String)> redirect stubs   ← NEW
7. template ║
8. write
```

New module `src/build/alias.rs` with:

```rust
pub fn run(
    config: &Config,
    site_data: &SiteData,
    index: &DocIndex,
) -> Result<Vec<(PathBuf, String)>>
```

It iterates the frozen index, and for each doc with a non-empty `aliases`,
computes the target URL once and renders one stub per alias. It can run in
parallel with Rayon over docs (like `archive::run`), since stubs are independent
and write to the side. In src/build.rs the result is concatenated onto `outputs`
before `write::run` — alias stubs are just more `(output_path, content)` pairs, so
`write.rs` needs **no change**.

(Aliases could alternatively be folded into the template phase, but a separate
stage keeps the "generated, never-classified" property explicit and matches the
archive precedent — alias stubs must never appear in `collection()`/`taxonomy()`/
backlinks/sitemaps.)

### Collisions & precedence

Three cases, resolved deterministically:

1. **Alias == its own canonical path** → skip silently (no self-redirect).
2. **Alias collides with a real page's `output_path`** → the real page wins; the
   alias is dropped with a `warn!`-style stderr message (`alias '/x/' from
   'a.md' shadowed by page 'b.md' — skipping`). A real page must never be
   clobbered by a redirect stub.
3. **Two docs claim the same alias** → deterministic winner (lexicographically
   smallest `id_path`, matching the wikilink tiebreak in
   [resolution rules]) plus a `warn!` naming both. Ambiguous, but we pick a stable
   result rather than letting write order decide.

Implementation: build a `HashSet`/`HashMap` of real `output_path`s from the index
first (case 2), then fold aliases left-to-right by sorted `id_path` (cases 1 & 3).
Emit warnings to stderr; never hard-error, since a stale alias shouldn't break a
build.

### Config

Zero new required config. The feature is always on; a doc with no `aliases` field
emits nothing. No `aliases_dir`, no flag.

(Optional, deferred — see Non-goals — a future `redirects` config block for
server-style `_redirects`/`_headers` output.)

## Edge cases

- **Empty / whitespace-only alias** → skip with a warning.
- **Alias with `..` or absolute-escaping segments** → reject (must stay inside
  `output_dir`); reuse whatever path-safety check `write` implies, or normalize
  and verify the result has no `..` components.
- **base_path set** → target is `base_path`-prefixed; the alias *file* is written
  at the literal alias path under `output_dir` (the alias represents the on-disk
  URL the host serves, which already includes any sub-path mount — consistent with
  how `static/` is copied verbatim). Document this explicitly so `/blog`
  deployments don't double-prefix.
- **`italic serve`** — stubs are real files in `output_dir`, so the dev server's
  live reload and the `<meta refresh>` both work unchanged. Worth one manual check
  that the injected live-reload script and a `0;` refresh don't race awkwardly
  (they shouldn't — refresh fires after load).
- **HTML/YAML docs** — aliases are read from frontmatter regardless of `DocKind`,
  so `.html` and `.yaml` documents support them too.

## Testing

Follow the existing `tests/fixtures/NN_name` golden-file convention (next free
index is `21_`):

- `21_aliases/` — a doc with two aliases (`/old/`, `/legacy.html`), asserting both
  stub files exist, contain the right `<meta refresh>` target, and the canonical
  page is unaffected. Exercises both the directory-append and literal-file paths.
- `22_alias_collision/` — two docs claiming one alias, plus an alias that collides
  with a real page; assert the deterministic winner and that the real page is
  untouched.
- Unit tests in src/permalink.rs for `alias_output_path` (trailing slash, no
  extension, `.html`, leading-slash stripping, `:`-in-path left literal),
  alongside the existing `expand`/`to_url` tests.
- A `--drafts` vs production test confirming a draft's aliases don't emit (likely
  folds into the existing drafts fixture `17_drafts`).

## Phasing

**Phase 1 (this proposal):** client-side redirect stubs as described. Small,
self-contained, no new dependencies.

**Phase 2 (deferred, separate proposal):** host-level redirect maps — emit a
Netlify `_redirects` / Cloudflare `_redirects` / `_headers` file from the same
`aliases` data (and possibly a dedicated `redirects:` config for bulk/wildcard
rules). Server redirects are faster and return real 301s, but they're
host-specific; the HTML stub is the universal baseline that works on any static
host, so it ships first.

## Non-goals

- **301 status codes** — impossible from a static file; that's Phase 2's
  host-level job. The `<meta refresh>` + `canonical` combo is the static-site
  standard and is what Hugo emits too.
- **Pattern expansion in aliases** (`:slug` etc.) — aliases are literal historical
  URLs.
- **Aliases on archive/generated pages** — only authored docs for v1. Archives can
  already control their own `permalink`; redirecting them is out of scope.

## Resolved decisions

- **Scope:** Phase 1 only (client-side HTML stubs). Host-level redirect maps are
  deferred to a separate Phase 2 proposal.
- **`noindex`:** stubs are `canonical`-only, no `robots: noindex` — matches Hugo's
  default so the old URL's ranking is passed to the canonical target rather than
  dropped.

## Open questions

1. Override context surface: is `{ target, target_absolute, title, page }` enough,
   or should the full `site`/`data` globals be available in `alias.html` for
   themes that want branded redirect pages? Cheap to include; probably should.
2. Warn vs. silent on collisions — proposal says warn-to-stderr and continue.
   Confirm that matches italic's house style for non-fatal build issues (it
   matches the unresolved-wikilink behavior, which logs and degrades).
